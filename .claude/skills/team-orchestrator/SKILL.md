# AI Team Orchestrator

Slack 작업 요청을 분석하고, 적절한 에이전트 팀을 동적으로 구성하여 병렬 처리하는 오케스트레이터.

## 실행 모드: 에이전트 팀 (감독자 패턴)

작업 유형에 따라 팀 구성을 동적으로 결정한다. 모든 작업이 동일한 팀을 쓰는 것이 아니라, 요청 내용에 맞는 전문가를 선택하여 팀을 구성한다.

## 에이전트 풀

| 에이전트 | 역할 | 투입 기준 |
|---------|------|----------|
| developer | 코드 작성, 프로젝트 생성, 버그 수정 | 코딩/개발 작업이 포함된 경우 |
| researcher | 웹 검색, 정보 수집, 트렌드 조사 | 조사/검색/리서치가 필요한 경우 |
| reviewer | 코드 리뷰, 품질 검증 | 코드 산출물이 있고 품질 검증이 필요한 경우 |
| analyst | 데이터 분석, 비교, 레포트 작성 | 분석/비교/요약이 필요한 경우 |

## 워크플로우

### Phase 0: 컨텍스트 확인

1. 요청과 관련된 기존 프로젝트 디렉토리(`~/a-projects/`)가 있는지 확인
2. `_workspace/` 존재 여부로 실행 모드 결정:
   - **미존재** → 초기 실행, Phase 1 진행
   - **존재 + 부분 수정 요청** → 해당 에이전트만 재호출
   - **존재 + 새 입력** → `_workspace/`를 `_workspace_{timestamp}/`로 이동 후 Phase 1

### Phase 1: 요청 분석 및 팀 구성 결정

1. Slack 요청 내용을 분석하여 작업 유형을 분류한다:

| 작업 유형 | 예시 | 투입 에이전트 |
|----------|------|-------------|
| 프로젝트 생성 | "XX 프로젝트 만들어줘" | developer + reviewer |
| 웹 조사/리서치 | "XX 조사해줘", "트렌드 알려줘" | researcher + analyst |
| 코드 개발 | "XX 기능 추가해줘", "버그 수정해" | developer + reviewer |
| 분석/비교 | "XX 비교해줘", "연봉 테이블" | researcher + analyst |
| 복합 작업 | "폴더 만들고 조사해서 레포트" | developer + researcher + analyst |

2. 단순 질문/인사("안녕", "상태")는 팀 구성 없이 직접 응답한다.

3. 작업 복잡도에 따라 팀 크기를 결정한다:
   - **단순** (단일 작업): 에이전트 1~2명
   - **중간** (다단계 작업): 에이전트 2~3명
   - **복합** (조사+개발+검증): 에이전트 3~4명

### Phase 2: 팀 구성 및 작업 할당

1. 팀 생성:
   ```
   TeamCreate(
     team_name: "{작업주제}-team",
     members: [선택된 에이전트들, 각각 model: "opus"]
   )
   ```

2. 작업 등록:
   ```
   TaskCreate(tasks: [
     각 에이전트에게 할당할 구체적 작업 목록,
     의존 관계가 있으면 depends_on 명시
   ])
   ```

3. 작업 지시 시 반드시 포함할 내용:
   - 작업 디렉토리 (`~/a-projects/{프로젝트명}/`)
   - 산출물 저장 경로 (`_workspace/{phase}_{agent}_{artifact}.md`)
   - 완료 기준
   - 한국어 사용 지시

### Phase 3: 팀 실행 및 모니터링

팀원들이 자체 조율하며 작업을 수행한다. 리더는:
- `TaskGet`으로 진행 상황을 확인한다.
- 팀원 유휴 알림 시 추가 작업을 할당하거나 지원한다.
- 팀원 간 데이터 요청은 `SendMessage`로 직접 처리하도록 허용한다.
- 막히는 팀원이 있으면 개입하여 가이드한다.

### Phase 4: 결과 통합 및 보고

1. 모든 팀원의 작업 완료를 `TaskGet`으로 확인한다.
2. 각 팀원의 산출물을 수집한다.
3. 결과를 통합하여 **Slack 응답용 요약**을 생성한다:
   - 핵심 결과를 먼저 (3줄 이내)
   - 상세 내용은 파일 경로로 안내
   - 3800자 이내로 요약 (Slack 메시지 제한)

### Phase 5: 정리

1. 팀원들에게 종료 알림
2. `TeamDelete`로 팀 정리
3. `_workspace/` 보존

## 데이터 흐름

```
[Slack 메시지] → [리더: 요청 분석]
                      ↓
              [팀 구성 결정]
                      ↓
         TeamCreate + TaskCreate
              ↓              ↓
        [에이전트A]    [에이전트B]
              ↓              ↓
        산출물 파일     산출물 파일
              └──────┬───────┘
                     ↓
           [리더: 결과 통합]
                     ↓
            [Slack 응답 전송]
```

## 에러 핸들링

| 상황 | 전략 |
|------|------|
| 팀원 1명 실패 | SendMessage로 상태 확인 → 재시작 또는 다른 팀원에게 재할당 |
| 팀원 과반 실패 | 현재까지 결과로 부분 보고, 실패 영역 명시 |
| 타임아웃 (30분) | 현재까지 수집된 부분 결과 사용, 미완료 부분 명시 |
| 데이터 충돌 | 출처 병기, 삭제하지 않음 |

## 테스트 시나리오

### 정상 흐름
1. Slack에서 "idea-search 폴더를 만들고 아이디어를 조사해줘" 수신
2. Phase 1: 복합 작업(개발+조사) 판단 → developer + researcher + analyst 선택
3. Phase 2: 팀 구성 (3명), 작업 할당 (프로젝트 생성 / 웹 조사 / 분석 레포트)
4. Phase 3: 팀원들이 병렬 작업 수행
5. Phase 4: 결과 통합하여 Slack에 요약 보고
6. Phase 5: 팀 정리

### 에러 흐름
1. researcher가 웹 검색 중 타임아웃
2. 리더가 감지 → SendMessage로 상태 확인
3. 재시작 시도 → 실패 시 analyst에게 가용 데이터로 분석 지시
4. 부분 결과로 보고서 생성, "일부 소스 미수집" 명시
